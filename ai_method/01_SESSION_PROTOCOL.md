# 01_SESSION_PROTOCOL

## Start of session

Every session should begin by identifying:

1. What is the user trying to accomplish?
2. Is the task simple or complex?
3. Does it require a spec?
4. Does it require a verifier?
5. Which files or context are needed?
6. What should not be done in this session?
7. What is the next minimal useful action?

## Session types

0 — method setup  
1 — project intake  
2 — corpus mapping  
3 — interview  
4 — spec drafting  
5 — verifier design  
6 — execution  
7 — review  
8 — skill capture  
9 — improve system  
10 — automation design

## Default sequence for complex work

1. Interview the user.
2. Draft or update SPEC.md.
3. Draft or update VERIFY.md.
4. Identify environment files.
5. Break work into small phases.
6. Execute one phase.
7. Verify.
8. Summarize uncertainty.
9. Capture durable learning.

## Non-submersion rule

Do not ingest an entire repository at once.

Start with:

- README;
- file tree;
- indexes;
- current notes;
- relevant specs;
- only then specialized files.

## Work chunk rule

Complex work should be split into:

- intake;
- map;
- decide;
- spec;
- verify;
- execute;
- review;
- capture.

Do not mix all phases into one response unless the user explicitly asks for a compact overview.

## End of session

End every substantial session with:

- what was established;
- what remains uncertain;
- what was changed;
- what should be verified;
- what should not be done next;
- next minimal action.
