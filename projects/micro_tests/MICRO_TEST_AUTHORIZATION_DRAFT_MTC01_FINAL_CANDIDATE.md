# MICRO_TEST_AUTHORIZATION_DRAFT_MTC01_FINAL_CANDIDATE

## Statut

Candidat finalisé pour une autorisation future éventuelle.

Ce document ne lance pas le micro-test.

Ce document ne crée pas de patch, mapping, prototype, spec globale ou engine.

Il transforme MTC01 en objet de décision propre.

---

# 1. Candidat

```text
MTC01 — Sub soutient didgeridoo sans le masquer
```

Question unique :

```text
Un sub peut-il soutenir le corps sans masquer les formants, le souffle et la pression organique du didgeridoo live ?
```

---

# 2. Décision actuelle

```text
MTC01 est un candidat prioritaire.
MTC01 n’est pas autorisé.
```

La seule décision possible plus tard sera :

```text
autoriser un micro-test limité ;
réviser encore ;
suspendre ;
abandonner.
```

---

# 3. Objet central

L’objet central du test n’est pas le sub.

L’objet central est :

```text
la lisibilité, la présence et l’agency du didgeridoo live.
```

Le sub est seulement une variable de soutien.

Si l’attention musicale bascule du didgeridoo vers le sub, le test signale un risque.

---

# 4. Ce qui serait testé

Seulement :

```text
la relation soutien / masquage entre didgeridoo live et fonction sub.
```

Le test ne chercherait pas un beau son final.

Le test ne chercherait pas un engine utilisable.

Le test ne chercherait pas un patch réutilisable.

---

# 5. Ce qui ne serait pas testé

Ne pas tester :

```text
architecture Vesperare ;
engine sub final ;
patch Max for Live ;
rack Ableton ;
mapping de performance ;
interface finale ;
scène complète ;
mix complet ;
relation aux guimbardes ;
relation au gong ;
relation au tambour ;
retour au corps global ;
spec globale ;
verifier complet.
```

---

# 6. Conditions minimales avant autorisation

Le micro-test ne pourrait être autorisé que si toutes ces conditions sont acceptées :

```text
1. Yohan confirme explicitement qu’il veut essayer ce micro-test ;
2. une seule question est maintenue ;
3. durée maximale : 30 à 45 minutes, installation comprise ;
4. nombre de variantes limité : 2 ou 3 états maximum ;
5. le didgeridoo live reste central ;
6. le sub reste accompagnement ;
7. le test inclut au moins un moment de retrait / réduction du sub ;
8. la charge mentale de Yohan est observée ;
9. aucun patch, preset, rack, routing ou réglage n’est conservé comme base ;
10. un résultat positif est traité comme indice local, pas validation du sub engine.
```

---

# 7. Variantes maximales autorisables

Si le test est autorisé plus tard, il ne devra pas dépasser :

```text
A — didgeridoo sans sub ;
B — didgeridoo + sub discret ;
C — didgeridoo + sub plus présent mais encore non dominant.
```

Pas d’autres sources.

Pas de guimbardes.

Pas de gong.

Pas de tambour.

Pas de scène complète.

---

# 8. Observations obligatoires si le test existe un jour

```text
Le didgeridoo reste-t-il lisible ?
Le souffle reste-t-il perceptible ?
Les formants restent-ils vivants ?
La pression organique est-elle soutenue ou remplacée ?
Le sub peut-il se réduire ou se retirer ?
Yohan reste-t-il dans le jeu ou sort-il en écoute technique ?
L’attention musicale reste-t-elle sur le didgeridoo ou bascule-t-elle vers le sub ?
```

---

# 9. Critères d’arrêt

Arrêter immédiatement si :

```text
le test devient un patch ;
il exige une architecture ;
il commence à définir un engine ;
il ajoute les guimbardes, gong ou tambour ;
il ajoute une interface ;
il cherche un beau son final ;
il masque le didgeridoo ;
il empêche Yohan de rester dans le jeu ;
il donne envie de conserver le montage comme base ;
il produit plus de décisions que d’observations.
```

---
