# MICRO_TEST_AUTHORIZATION_DRAFT_MTC01

## Statut

Brouillon d’autorisation pour un micro-test potentiel.

Ce document n’autorise pas encore l’exécution du test.

Il définit les conditions qui devraient être vraies avant de lancer le micro-test.

---

# 1. Micro-test candidat

```text
MTC01 — Sub soutient didgeridoo sans le masquer
```

Question unique :

```text
Un sub peut-il soutenir le corps sans masquer les formants, le souffle et la pression organique du didgeridoo live ?
```

---

# 2. Pourquoi ce candidat est prioritaire

Ce candidat touche le risque central identifié jusqu’ici :

```text
le sub peut soutenir l’agency live,
mais il peut aussi déplacer le centre musical hors du geste de Yohan.
```

Le test ne chercherait pas à créer un sub engine.

Il chercherait seulement à observer une relation :

```text
soutien versus masquage.
```

---

# 3. Dimension d’agency concernée

```text
souffle ;
corps ;
formants ;
pression organique ;
lisibilité du live ;
relation sub/didgeridoo ;
non-remplacement.
```

---

# 4. Ce qui serait testé

Seulement :

```text
la possibilité qu’une pression grave synthétique ou paramétrique soutienne le didgeridoo sans masquer sa présence live.
```

Observation attendue :

```text
le sub aide / le sub masque / résultat ambigu / test mal posé.
```

---

# 5. Ce qui ne serait pas testé

Ne pas tester :

```text
architecture du futur système ;
engine sub final ;
patch Max for Live ;
mapping de performance ;
interface finale ;
scène complète ;
mix complet ;
relation aux guimbardes ;
relation au gong ;
relation au tambour ;
retour au corps global ;
spec Vesperare.
```

---

# 6. Conditions minimales avant autorisation

Le test ne pourrait être autorisé que si :

```text
1. une question unique est maintenue ;
2. la durée est limitée ;
3. le résultat attendu est une observation, pas un patch ;
4. le didgeridoo live reste l’objet central ;
5. le sub est considéré comme variable d’accompagnement ;
6. aucun mapping durable n’est créé ;
7. aucun fichier central du projet ne dépend du résultat ;
8. le test peut être jeté ;
9. Yohan valide explicitement que le test vaut la peine ;
10. le résultat est ensuite classé comme indice, pas comme décision.
```

---

# 7. Critères d’arrêt

Arrêter immédiatement si :

```text
le test devient un patch ;
il exige une architecture ;
il commence à définir un engine ;
il ajoute les guimbardes, gong ou tambour ;
il ajoute une interface ;
il cherche un beau son final ;
il masque le didgeridoo ;
il donne envie de conserver le montage comme base ;
il produit plus de décisions que d’observations.
```

---

# 8. Format de résultat si le test est un jour autorisé

```text
Date :
Question unique :
Matériel utilisé :
Durée réelle :
Observation principale :
Le didgeridoo reste-t-il lisible ?

Résultat : utile / ambigu / mauvais / test mal posé
Décision : abandonner / refaire / transformer en claim / transformer en critère de verifier
Ce qui ne doit pas être réutilisé :
```
