# METHOD_COHERENCE_CHECKLIST

## Purpose

Use this checklist to determine whether the installed AI operating method is coherent enough to use.

## Checklist

### Core method

- [ ] The method includes Spec.
- [ ] The method includes Verifier.
- [ ] The method includes Environment.
- [ ] The method includes Review.
- [ ] The method includes Skill capture.
- [ ] The method includes Improve system.

### Practical operation

- [ ] There is a session protocol.
- [ ] There is a quickstart.
- [ ] There is a branch safety rule.
- [ ] There is a method status script.
- [ ] There are templates.
- [ ] There are checklists.

### Skills

- [ ] project-intake exists.
- [ ] claim-register exists.
- [ ] acquis-audit exists.
- [ ] write-spec exists.
- [ ] verify-output exists.
- [ ] ingest-source exists.
- [ ] improve-system exists.
- [ ] automation-check exists.
- [ ] ask-board exists.
- [ ] internal-focus-group exists.
- [ ] method-review exists.

### Safety

- [ ] Main is protected by rule.
- [ ] Experimental method work happens on a separate branch.
- [ ] Human validation zones are explicit.
- [ ] Automation is not default.
- [ ] Prompt rules are not mistaken for hard security.

### Existing project restart

- [ ] Existing docs are treated as corpus, not proof.
- [ ] Project intake precedes audit.
- [ ] Claim register precedes acquis audit.
- [ ] Spec reconstruction comes after claim audit.
- [ ] No prototype is implied by documentation maturity.

### Human judgment

- [ ] Human taste is preserved.
- [ ] Domain judgment is preserved.
- [ ] AI confidence is not treated as proof.
- [ ] Simulated perspectives do not replace real validation.

## Failure conditions

Reject the method if:

- it encourages immediate execution from vague prompts;
- it lacks verifier logic;
- it lacks branch safety;
- it treats existing documentation as truth;
- it automates judgment-heavy work by default;
- it cannot guide the next real session.

## Output

PASS / NEEDS REVISION / REJECT
