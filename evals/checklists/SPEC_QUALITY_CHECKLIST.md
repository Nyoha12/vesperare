# SPEC_QUALITY_CHECKLIST

## Purpose

Use this checklist before accepting any SPEC.

A SPEC is not valid because it is detailed.  
It is valid if it reduces ambiguity and makes execution verifiable.

## Checklist

- [ ] The real goal is explicit.
- [ ] The user / audience is explicit.
- [ ] Non-goals are listed.
- [ ] Constraints are listed.
- [ ] Assumptions are visible.
- [ ] Key decisions are listed.
- [ ] Alternatives are mentioned for important decisions.
- [ ] Definition of done is concrete.
- [ ] Verification is defined before execution.
- [ ] Human validation zones are identified.
- [ ] The spec is scoped small enough for action.
- [ ] The spec does not hide uncertainty.
- [ ] The spec does not turn hypotheses into decisions.
- [ ] The spec does not authorize implementation prematurely.

## Failure conditions

Reject the spec if:

- it is mostly a narrative;
- it lacks verification;
- it contains vague goals such as "make it better";
- it assumes user intent without interview;
- it makes irreversible decisions without human validation.

## Output

PASS / NEEDS REVISION / REJECT
